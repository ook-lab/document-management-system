## できていること（中級以上の証拠）

### 1. 実用的なシステムを動かしている

17のマイクロサービスをCloud Runにデプロイし、自分の生活・業務課題を技術で解決している（PDF処理、Gmail連携、Google Drive連携、家計簿、転職ツール、ネットスーパー検索など）。チュートリアルをなぞったレベルではない。

### 2. アーキテクチャの設計判断ができている

- Web API（enqueue/searchのみ）とWorker（処理専用）の明確な責務分離
- `shared/` と `services/` の依存方向制約（shared → services のインポート禁止）
- SSOTの概念を運用CLI（ops.py）やパイプラインに適用
- dry-run → apply の二段階実行による事故防止

### 3. パイプライン設計に独自の工夫がある

- Stage A の `A5TypeAnalyzer` がYAMLルールベースでページ単位の信頼度マップ（`page_confidence_map`）を生成し、LOW信頼度ページはBをスキップしてEのOCRに回すルーティング制御
- B-90（LayerPurge）でテキスト白塗り → AIの二重読み取り防止
- Stage E の文字密度でAIモデル切替（Flash-lite vs Flash）によるコスト最適化
- E32で「候補集合に止める」→E40で「controllerが明示実行」という制御フロー
- Stage D の OpenCV モルフォロジー + Hough 検出にスキャン画像の「線爆発」防止ガード

### 4. 運用を見据えた設計ができている

- `ops.py`（845行）：`ops_requests`テーブルがSSOT、`worker_state.stop_requested`は派生キャッシュという正しい状態管理パターン
- リース管理、ステータスリセット、ステージクリアなど運用操作を一通り装備
- Cloud Buildトリガーの管理スクリプト、デバッグパイプライン等の運用ツール
- RLS（Row Level Security）をWorkerとWeb APIで使い分け

### 5. GCPデプロイが成熟している

- サービスごとの `cloudbuild.yaml`、Secrets Manager連携
- `includedFiles` によるトリガーの精密制御
- デプロイREADMEでのアンチパターン記録（実際の障害から学んだ対策）

### 6. AIツールの制御ができている

- CLAUDE.md や Cursor ルールで AI の振る舞いを制御
- 「責任をそらす言い方をしない」というトーン指定（AIとの協働の失敗から学んだ結果）

## 弱点・伸びしろ

### 1. テストがゼロ

`tests/` ディレクトリが空。877コミット中、test関連コミットなし。CLAUDE.md に pytest コマンドがあるが対象が存在しない。

### 2. コミット品質にばらつき

Conventional Commits（`feat(`, `fix(` 等）は 877 中わずか 70 件（約8%）。`assddf` のような意味不明コミットが混在。

### 3. コードの重複

`services/doc-processor/scripts/` と `scripts/` にほぼ同じファイルが二重に存在。リファクタリング途中で止まった形跡。

### 4. GitHub Actions の一部が壊れている

- `hourly_monitor.yml` が参照する `scripts/inbox_monitor.py` は実際には `services/data-ingestion/monitoring/` にある
- `manual_check.yml` が参照する `scripts/check_llm_setup.py` は存在しない

### 5. Stage H が未完成

`__init__.py` が `StageH1Table` / `StageH2Text` をインポートしようとしているが対応ファイルが存在しない。`PipelineManager` のメインパスでは H を飛ばしている。

### 6. フロントエンドが未成熟

`frontend/` ディレクトリがない。Stage G で UI 最適化を行っているがそれを受けるフロントの存在がリポジトリ上で見えない。

## 総括

5.5ヶ月で17サービス・877コミットのシステムをAIと協働で構築・運用しているのは相当な実行力。PDFの物理構造解析からAI抽出、UI最適化までの一貫したパイプライン設計は、ドメインの問題を深く理解しないと作れない。

「AIを使って作っている」ことは弱みではなく、現代のソフトウェア開発における正当なスキル。次のレベルに行くためには、AIに任せている部分のうち「自分でも判断できる」範囲を広げていくことが鍵。テストを書く、設計判断を言語化する、小さなモジュールを自力で書いてみる——この3つが最もレバレッジの高い次のステップ。
